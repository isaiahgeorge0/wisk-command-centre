# WISK — Product Principles

These principles guide every product, design, 
and technical decision made on WISK. When facing 
a difficult decision, check it against these 
principles first.

## 1. Daily use is the only measure that matters

A feature that impresses in a demo but is never 
used daily has failed. Every feature must earn 
its place by being genuinely useful on a regular 
basis. If users are not opening WISK every day, 
we have not solved the right problems yet.

## 2. Friction kills habits

The hardest part of any productivity tool is 
getting people to use it consistently. Anything 
that adds unnecessary steps, requires too much 
setup, or slows down a common action is a threat 
to the product. Speed and simplicity of data 
entry are non-negotiable.

## 3. Surface insights, not just data

Storing information is not enough. WISK must 
help users understand what their information 
means and what to do next. A list of tasks is 
useful. Knowing which task is most critical 
right now is powerful. Always ask: what does 
this data tell the user, and are we showing 
them that?

## 4. Everything is connected

A lead should be able to become a project. A 
project generates tasks. Tasks connect to goals. 
Goals drive content. Content generates leads. 
The connections between things are where WISK 
creates value that no single-purpose tool can 
match. When building a feature, always consider 
how it connects to everything else.

## 5. The core must work perfectly before 
packages are added

The core product — projects, tasks, goals, 
ideas, calendar, leads, content — must be 
genuinely excellent before any package is 
built. A property package built on a weak 
foundation is worse than no property package. 
Packages amplify what is already there. They 
do not compensate for what is missing.

## 6. Packages solve specific problems for 
specific people

Every package must solve a clearly defined 
high-value problem for a clearly defined type 
of user. Vague packages that try to serve 
everyone serve no one. Before building a 
package, define exactly who it is for and 
exactly what problem it solves.

## 7. AI must be explainable and trustworthy

AI features in WISK must earn the user's trust. 
An AI that says "your website audit reel 
outperformed your average by 42% because the 
hook was direct and the pacing was quicker" 
is trustworthy. An AI that says "post more 
videos" is not. Every AI output must explain 
its reasoning.

## 8. Human approval before consequential actions

Agents and automation should never take 
consequential actions without human review. 
Sending emails, modifying production data, 
publishing content, charging clients — all 
require explicit approval. Automation should 
reduce effort, not remove oversight.

## 9. Mobile is a companion, not a replacement

WISK is a desktop-first product. The depth of 
information and the complexity of management 
tasks are best suited to a large screen. Mobile 
should provide quick access to the most 
important daily information — tasks, 
notifications, a quick capture for ideas — 
without trying to replicate the full desktop 
experience.

## 10. Exclusivity builds trust

WISK is invite-only for a reason. Early users 
should feel like they have access to something 
special. The quality of the early community 
shapes the product. Growth should be deliberate 
and quality-controlled, not viral and 
unmanaged.

## 11. Build for the ambitious, not the average

Every design and product decision should be 
made with an ambitious, capable user in mind. 
Do not dumb things down. Do not hide complexity 
behind excessive simplicity. Trust that users 
are intelligent and capable of using a powerful 
tool if it is well designed.

## 12. Subscriptions should feel like unlocks, 
not paywalls

Packages should feel like exciting additions 
that unlock new capability, not walls that 
block features the user already expects. The 
core product must be genuinely valuable for 
free or at base price. Packages are for users 
who want to go deeper in a specific area.

## 13. Consistency over novelty

WISK should feel like one coherent product, 
not a collection of features built by different 
people at different times. Every section should 
follow the same patterns, use the same 
components, and feel like it belongs. When in 
doubt, follow what already exists rather than 
inventing something new.

## 14. Feedback is infrastructure

User feedback is not a nice-to-have. It is 
how WISK improves. The feedback inbox, the 
admin panel, the personal DMs to early users — 
these are as important as any feature. A WISK 
that users love but never tell us about is a 
WISK that does not improve fast enough.
